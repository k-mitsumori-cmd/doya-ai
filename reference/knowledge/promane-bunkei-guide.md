## 1. そもそもドヤプロマネとは何か

### 一言でいうと
**「案件・タスク・収支を一元管理する小さな会社向けのプロジェクト管理ツール」** です。

### 何ができるか
- **ガントチャート**: プロジェクト全体の進捗を視覚化
- **カンバンボード**: タスクをドラッグ&ドロップで進捗管理（未着手→進行中→レビュー→完了）
- **時間記録 → 人件費自動算出**: 「メンバーの時給 × 作業時間」で原価が自動計算
- **収支管理**: 売上 − 原価（人件費+経費）= 利益 をリアルタイム可視化（**黒字/赤字**で色分け）
- **クライアント管理**: 顧客情報と案件の紐付け
- **メンバー招待**: 招待リンク（**メール認証必須**、30日有効）
- **複数ワークスペース対応**: 個人用・会社用・部署用などを切替可能
- **レポート**: Recharts による月別/プロジェクト別グラフ

### 開発期間と工数
- 着手から本番リリースまで: **約3日**
- 修正・改善のラリー: **約50回** (バグレポート → 修正のループ多数)
- 作業した人: **1人（事業担当・コード書けない）**
- 主に使ったAI: **Claude（Anthropic社）**

### 何が「すごい」のか
- 普通なら「3人 × 6ヶ月 = 1,800万円」レベルのSaaSを、AI活用で**1人 × 3日 = ほぼ実費のみ**で実現
- 招待メール配信、Stripe決済連携、セキュリティ対策まで含めた**実用レベル**
- バグが出ても **「修正してください」+ エラースクショ** で次々解消

### NextAuth Google OAuth の設定（必須）
ドヤプロマネは Google ログイン必須なので、これだけは事前準備が必要:

1. https://console.cloud.google.com/ で新規プロジェクト作成
2. 「APIとサービス」→「認証情報」→「OAuth 2.0 クライアント ID 作成」
3. 承認済みリダイレクト URI に `https://your-app.vercel.app/api/auth/callback/google` を追加
4. 取得した Client ID / Secret を Vercel の環境変数に設定:
   - `GOOGLE_CLIENT_ID`
   - `GOOGLE_CLIENT_SECRET`
   - `NEXTAUTH_SECRET` (ランダムな32文字)
   - `NEXTAUTH_URL` (本番URL)

これだけは Claude に頼んでも代わりにやってもらえないので、自分でやる必要あり。

## 5. 実際にハマった12のポイントと解決法

### ① Toast が全部「無音」だった
**症状**: バグ修正したはずなのに「サイレント失敗」報告が続く。エラーtoastも成功toastも見えない。

**真の原因**: 2つの toast ライブラリ (`sonner` と `react-hot-toast`) が混在していて、コンポーネントは `sonner` を使っているのに `<Toaster>` は `react-hot-toast` しか設置されていなかった。

**修正**: layout に `<SonnerToaster>` を追加するだけ。

**学び**: **「動かない原因がコード見ても分からない」時はライブラリの設定漏れを疑う**。Claude に「toast が表示されない原因を全部洗い出して」と聞くと気付ける。

---

### ② 利益率が「99.2%」「103%」と画面によってバラバラ
**症状**:
- ダッシュボード: 99.2%
- プロジェクト一覧: 103%
- レポート: 102.5%

**真の原因**:
- 過去のテストデータに「経費 -¥37,500」のような負値が残っていた
- 利益 = 売上 − 経費 だが、経費が負だと 利益が売上より大きくなり、利益率が100%超え
- ダッシュボードは修正したが、他画面に修正が反映されていなかった

**修正**:
1. 全画面で経費を `Math.max(0, expense)` でクランプ
2. 利益率を `Math.min(100, Math.max(-100, rate))` で範囲制限
3. 既存不正データを修復するボタンを設定ページに追加

**学び**: **「同じデータを複数画面で表示する」時は必ず計算ロジックを共通化**。あと過去データの汚染も考慮。

---

### ③ Server Action で「サーバーコンポーネントレンダーエラー」連発
**症状**: 顧客追加・経費追加・タスク追加など、複数の操作でボタンが固まりエラー。

**真の原因**: Next.js の Server Action は便利だけど、内部で `redirect()` を呼ぶとエラーになる仕様があった。それを使っていた認証関数 `requirePromaneAuth` で発生。

**修正**:
1. Server Action 用の `requirePromaneAuthAction()` を別途作って、エラーは `throw Error` で返す
2. さらに重要な操作は Server Action を捨てて **API ルート化**:
   - `/api/promane/clients` (顧客追加)
   - `/api/promane/expenses` (経費追加)
   - `/api/promane/members/[id]/rate` (時給更新)

**学び**: **Server Action は便利だけど落とし穴も多い**。複雑な操作・エラーハンドリングが必要なら最初から API Route のほうが安定。

---

### ④ 顧客追加が 5ラウンド連続で失敗
**症状**: 「追加する！」ボタンを押しても「追加中...」のまま戻らず、コンソールには「Unknown argument 'address'」のエラー。

**真の原因**: フォームから `address` フィールドを送信していたが、Prisma スキーマには `address` カラムが定義されていなかった。

**修正**: スキーマに `address String?` を追加して `npx prisma db push`。

**学び**: **「Prisma のエラーメッセージ Invalid invocation Unknown argument」は10割スキーマと送信データの不整合**。エラーメッセージで即解決できる典型例。

---

### ⑤ 1人の入力ミスで全員のダッシュボードが死ぬ
**症状**: テストで負の契約金額や逆転日付（終了日 < 開始日）を入れたら、ワークスペース全員のダッシュボードが永続的にエラー画面に。

**真の原因**:
- 集計関数が NaN / Infinity を計算してクラッシュ
- 一度クラッシュすると以降のページ生成も全部 500 エラー

**修正**:
1. 全集計を `safeNum()` で防御 (`null`/`NaN`/`Infinity` を 0 に)
2. `Promise.all` + 個別 try-catch で 1つ落ちても他は表示
3. `error.tsx` で「もう一度試す」ボタンを実装
4. 入力時に Zod 風バリデーション

**学び**: **本番サービスは「変なデータが入っても壊れない」を最優先**。1人の悪意/ミスでサービス全体が止まらない設計。

---

### ⑥ IDOR 脆弱性 11件
**症状**: なし。コードレビューで判明。

**真の原因**: 「workspace 所属確認」はしていたが、「リソース ID 自体が自分の workspace に属するか」確認していなかった。

**例**:
```typescript
// ❌ 危険: clientId が他人のかもしれない
await prisma.promaneClient.update({ where: { id: clientId }, data: {...} })

// ✅ 安全
const existing = await prisma.promaneClient.findFirst({
  where: { id: clientId, workspaceId: workspace.id }
})
if (!existing) throw new Error("見つかりません")
```

**修正**: 全 update/delete に `findFirst({ id, workspaceId })` の所属確認を追加。Client / Task / Project / TimeEntry / Expense / Member の 11箇所。

**学び**: **「セキュリティは専門家でないと無理」と思いがちだが、Claude のセキュリティレビュー機能を使えば素人でも8割カバーできる**。最後に "/security-review" 必須。

---

### ⑦ 招待リンクが漏れたら他人が参加できた
**症状**: コードレビューで判明。

**真の原因**: 招待トークンの所有者確認をせず、ログインしていれば誰でも承諾可能だった。

**修正**: 招待されたメールアドレスと Google ログインのメールが一致しないと 403 で拒否。

**学び**: **招待・パスワードリセット系は必ずメール一致確認**。これ漏れると「URLが流出 = 不正参加可能」。

---

### ⑧ 時間単価「-3000」が「+3000」として保存される（4ラウンド未解決）
**症状**: メンバーの時給に `-3000` を入力すると、エラーにならず「成功！」が出て `¥3,000` として保存。

**真の原因**: 自分の `onChange` 実装にバグがあった:

```typescript
// ❌ バグの原因
onChange={(e) => {
  const v = parseInt(e.target.value);
  setRate(Number.isFinite(v) ? v : 0);  // ← この「: 0」
}}
```

ユーザーが `-3000` を一文字ずつ入力すると:
1. `-` 入力 → `parseInt("-")` = NaN → `setRate(0)` → input表示が "0" にリセット
2. `0` が表示された状態で `3` 追加 → `"03"` → parseInt=3
3. `0` 追加 → `"030"` → parseInt=30
4. ... → 最終的に `"03000"` → parseInt=3000 として保存

**修正**: state を `number` ではなく `string` で保持。マイナス記号がそのまま残る。

```typescript
const [rateInput, setRateInput] = useState("0")
const rate = parseInt(rateInput)
// rate < 0 で UI を赤化、Save ボタン無効化、保存時 toast.error
```

**学び**: **React の controlled input で number 型 state を使うと罠が多い**。特に負数・小数点・先頭ゼロ等の入力中の中間状態を考慮。**string で保持 → 最後にparse** が安全。

---

### ⑨ Sonner Toaster がどこにもなくて 30回くらい混乱
**症状**: バグ修正の効果がユーザーに見えない。「修正したつもり」と「動いていない」の食い違いが何ラウンドも。

**真の原因**: `import { toast } from 'sonner'` を全コンポーネントで使っていたが、`<SonnerToaster>` コンポーネントを layout に置き忘れていた。`react-hot-toast` の Toaster はあったので、toast.error が呼ばれてもどこにも表示されない。

**修正**: layout に1行追加するだけ。

```tsx
import { Toaster as SonnerToaster } from "sonner"
...
<SonnerToaster position="top-center" richColors closeButton />
```

**学び**: **「全部修正したのに直らない」時は表示層の問題を疑う**。toast が動かない、router.refresh が効かない、router.push が来ない等。

---

### ⑩ 「window.confirm()」の見た目がダサくて統一感がない
**症状**: ブラウザネイティブの確認ダイアログ ("OK / キャンセル") が出る箇所が、モダンな UI と不釣り合い。

**修正**: `useConfirm()` hook を自作。全削除操作で統一されたカスタムモーダル（キャラクター画像 + グラデーションCTA）に置換。

**学び**: **OS依存の UI 部品（confirm/alert/prompt）は早めにカスタム実装に置換**。BtoB SaaSの体験品質を左右する。

---

### ⑪ ブラウザキャッシュで「修正してない」と勘違い 何度も
**症状**: コードに修正を入れて Vercel デプロイ済みなのに、ユーザーは「直ってません」と報告。

**真の原因**: 旧 JS バンドルがブラウザにキャッシュされていて、ハードリロードしないと反映されない。

**修正**:
1. サイドバー下に **「v.bbfa596」のようにビルドコミットハッシュ表示**
2. ユーザーが「今のバージョン何？」と確認できる
3. 古ければハードリロード or シークレットウィンドウへ誘導

**学び**: **「キャッシュかコードか」を切り分ける手段は必須**。バージョン表示があると Claude も判断しやすい。

---

### ⑫ Claude が「修正済み」と言うのに実態が違う
**症状**: Claude が「全部修正完了」と言っても、ユーザー実機テストで残バグ。

**真の原因**:
- コードレベルでは正しいが、デプロイ反映前にユーザーが触っていた
- ブラウザキャッシュで古いバージョンを見ていた
- 既存の DB データに「不正な値」が残っていて、それが集計を歪めていた

**修正**:
- Claude に「次の deploy が反映されるまで 3分待ってください」と伝言依頼
- DB の既存不正データを一括修復するボタン（`/api/promane/repair`）追加
- バージョン表示で確認可能に

**学び**: **「コード修正 ≠ ユーザー体験の修正」**。デプロイ反映・キャッシュクリア・既存データの修復、までセットで考える。

## 6. 文系でも分かる用語集

| 用語 | 平たく言うと |
|---|---|
| **Server Action** | Next.js でクライアントから直接サーバー関数を呼べる仕組み（便利だが落とし穴多い） |
| **API Route** | URL を叩いてサーバー処理する従来型（安定） |
| **Prisma** | データベースを TypeScript で扱える「ORM」（オーアールエム） |
| **NextAuth** | Google や GitHub ログインを簡単に実装するライブラリ |
| **IDOR** | 他人のデータを ID 直指定で見れちゃう脆弱性 (例: `/user/123` を `/user/124` に変えて他人の情報を見る) |
| **Server Components renderエラー** | Next.js でサーバーレンダリング中に何かが落ちた時のエラー（原因が見えづらい） |
| **Recharts** | グラフ描画ライブラリ（折れ線・棒・円グラフ等） |
| **@dnd-kit** | ドラッグ&ドロップを実装するライブラリ |
| **Resend** | メール送信 SaaS（月3,000通まで無料） |
| **キャッシュ** | 同じデータを毎回取りに行かず近くに保管しておく仕組み (古い情報が出る原因) |
| **Vercel デプロイ** | コードを本番サーバーに反映する作業 (git push で自動) |
| **ワークスペース** | チームや組織を分ける単位 (≒ Slack のワークスペース) |
| **role (権限)** | owner / admin / member / guest で誰が何できるか制御 |
| **RBAC** | Role Based Access Control。役割ベースの権限管理 |
| **Hourly Rate** | 時給。人件費計算に使う |

## 8. 次に作るなら気をつけたいこと

### ✅ 最初にやるべきこと
1. **既存サービス3つ触る**: Notion / Asana / Backlog をユーザー目線で観察
2. **データモデルから先に決める**: 後から大幅変更すると面倒
3. **Google OAuth は最初に設定**: 後回しにすると詰む
4. **モックアップ画像を ChatGPT で作る**: UI 方向性のブレ防止

### ⚠️ よくある落とし穴
1. **「Server Action でいけるやろ」と思った機能が SC エラーで死ぬ** → 最初から API Route で実装が安心
2. **toast ライブラリの混在** → 最初から1本に決める (`sonner` か `react-hot-toast` か)
3. **入力バリデーション後回し** → サーバー側 + クライアント側の両方必須
4. **セキュリティ後回し** → 最初から IDOR / メール認証 / RBAC を意識
5. **「過去のテストデータ」が本番に残る** → 修復スクリプトを用意
6. **キャッシュで「直ってない」と勘違い** → バージョン表示を必ず付ける

### ✨ Claude を使う時のコツ
1. **連続報告には番号を付ける**: 「① ② ③」で構造化すると修正漏れない
2. **スクショ + 文章で伝える**: 言葉だけより圧倒的に伝わる
3. **「セキュリティレビュー」を最後にやる**: `/security-review` で IDOR / XSS / CSRF を自動検出
4. **デプロイ反映タイムラグを意識**: コード修正後 3分待ってからテスト
5. **「全部修正してください」は通じる**: 並列で対応してくれる

### 💡 ドヤプロマネ特有のノウハウ
1. **ガントチャート + カンバン**: 同じタスクを2視点で見せると体験が大幅向上
2. **収支計算は黒字/赤字で色分け**: 数字だけより一瞬で理解できる
3. **招待リンク**: メール一致確認は絶対 (漏洩しても本人以外参加不可)
4. **複数ワークスペース**: 個人/会社/部署で切替できると BtoB で評価が上がる
5. **データ修復ボタン**: 不正データが入っても1クリックで正常化できる安心感

## まとめ

**「コードが書けない=作れない」は、もう古い時代の話です。**

Claude のような AI アシスタントを「優秀な開発パートナー」として活用すれば、
文系・非エンジニアでもドヤプロマネレベルの実用BtoB SaaSが3日で作れます。

ただし大事なのは:
1. **エラーを丁寧に Claude に伝える**（スクショ + 文章）
2. **「修正済み」を鵜呑みにせず実機テスト**
3. **キャッシュ問題を意識**（バージョン表示で確認）
4. **セキュリティは最後に必ず /security-review**

これだけでプロダクトの品質は大きく上がります。**まずは小さく始めてみてください**。
