## 1. そもそもドヤリストとは何か

### 一言でいうと
**「業界・地域・規模を選ぶだけで、AIが自動で営業リストを作ってくれるツール」** です。

### 何ができるか
- 業界（IT/製造業/広告など14業界）と地域（全国/関東/関西など）を選ぶ
- 最大10,000社の企業データを自動で抽出
  - 法人名・住所・都道府県・法人番号
  - 代表者名・従業員数・資本金・公式URL（登録ありの企業のみ）
- そのリストに対して**営業メール文面・フォーム文面・電話スクリプト**もAIが自動生成
- AI キーワード変換（「DX」→「デジタル/システム/IT」等の検索ワード自動展開）
- CSV/Excelでダウンロードして即営業に使える

### 開発期間と工数
- 着手から本番リリースまで: **約2日**
- 修正・改善のラリー: **約30回**
- 作業した人: **1人（営業/事業担当・コード書けない）**
- 主に使ったAI: **Claude（Anthropic社）**

### 使ったデータソース（全部無料）
- **国税庁 法人番号公表サイト**: 約530万社の基本情報
- **gBizINFO API（経済産業省）**: 詳細データ（代表者・資本金等）
- **Google Gemini API**: AI文章生成・キーワード変換

### 何が「すごい」のか
- 普通なら「3人 × 3ヶ月 = 900万円」レベルの開発を、AI活用で**1人 × 2日 = ほぼ実費のみ**で実現
- コードを1行も書かずに、すべて自然言語の指示で完成
- 「動くだけ」じゃなく、デザイン・セキュリティ・課金まで実用レベル

## 5. ドヤリスト開発でハマった10のポイントと解決法

### ① gBizINFO API の制限を読まずに作ったら 1500社で打ち切り
**症状**: 10,000社を指定したのに 1,494社しか取れない。

**真の原因**:
- gBizINFO は 1ページ最大 limit=5000 だが、page は最大10
- 当初 limit=50 × page=10 = 500件/キーワード × 3キーワード で重複排除すると約1,500社が天井だった

**修正**: limit を 50→1000 に変更。1000 × 10 = 10,000件/キーワード × 3キーワード = 約2万社のプールから抽出可能に。

**学び**: **「公式ドキュメントを鵜呑みにせず、実際に試す」** が最重要。Claude に「APIの実際の上限を実測して教えて」と頼むとテストコードを書いて実行してくれる。

### ② 都道府県を漢字で渡したら 404
**症状**: 「東京都・製造業・100社」で抽出すると HTTP 404 が返ってアプリ画面が真っ白に。

**真の原因**: gBizINFO API の `prefecture` パラメータは **2桁の都道府県コード（JIS X 0401）が必須**。「東京都」と漢字で渡すと 404。
- ✅ `prefecture=13` (東京都のコード)
- ❌ `prefecture=東京都`

**修正**: 都道府県漢字 → 2桁コード変換マップを追加。エリア指定（関東・近畿等）も対応都道府県に展開して順次検索。

**学び**: APIの引数仕様は「文字列なら何でもOK」と思いがち。**実際にcurlで叩いて挙動を確認**するのが大事。

### ③ 「指定なし」が画面に出て恥ずかしい
**症状**: 従業員数欄に「指定なし 名」と表示された企業があった。

**真の原因**: ユーザーが企業規模フィルタで「指定なし」を選んだ値が、実データの代わりに表示されていた。

**修正**:
```typescript
function isValidEmployees(v) {
  if (!v || v === '指定なし') return false
  return /\d/.test(v)  // 数字を含む時だけ表示
}
```
表示する前にバリデーション関数を通す。

**学び**: フォールバック値（実データがない時の代替値）をそのままUIに出すと、変な表示になる。**「これって人間に見せて大丈夫？」と一度立ち止まる。**

### ④ 「推定 約2社」と表示されたバグ
**症状**: ドヤリストの右パネルに「該当する企業数（推定）: 約2社」と表示されたが、実際は5,000社以上ある。

**真の原因**: 推定APIで `data.totalCount` を取得しようとしたが、gBizINFO はそのフィールドを返さない。結果 `(空)+(空)=undefined` で 0 になり、UIには「1+1=2」と誤計算された値が表示された。

**修正**: 実際にデータを取得して件数をカウントする方式に変更。`limit=1000` で取得し、1000件返ってきたら「1000+」、10ページ目も1000件なら「10,000以上」と表現。

**学び**: APIレスポンスの「想定形式」と「実形式」は違うことがある。**console.log で実際のレスポンスを目視確認**するのが大事。

### ⑤ 業種コード「115」がそのまま表示
**症状**: 業種欄に「115」とだけ表示された企業があった。

**真の原因**: gBizINFO が稀に業種コード（JSIC = 日本標準産業分類）を数字で返すことがあり、それを変換せずに表示していた。

**修正**: 数字のみの値は表示しない（無効扱い）。
```typescript
function isValidIndustry(v) {
  if (!v || /^\d+$/.test(v)) return false
  return true
}
```

**学び**: ユーザー向けに表示する前に「これは人間が読めるか？」と一度フィルタリングする。

### ⑥ プラン表示が画面ごとに違う
**症状**: ヘッダーは「プロプラン」表示、料金ページは「無料プラン」表示で矛盾。

**真の原因**: 2箇所で別々のロジックでプラン名を取得していた。
- ヘッダー: `usage.plan.tier` を取得
- 料金ページ: `String(usage.plan).toUpperCase()` ← オブジェクトを文字列化していた

**修正**: 共通の `normalizePlan()` 関数を作って両方で使う。
さらに NextAuth セッションとAPIの両方から取得し、より上位のプランを採用（ダウングレード誤表示防止）。

**学び**: 同じデータを複数箇所で表示する時は、必ず共通関数化。Claudeに「これ DRY 原則違反かも」と言うと整理してくれる。

### ⑦ 履歴のタブ切替が機能しない
**症状**: 履歴ページで「フォーム文面」「メール文面」タブをクリックしても、リストだけが表示され続けた。

**真の原因**: ツール生成結果が DBに保存されていなかった。タブ切替自体は動いていたが、表示するデータがなかった。

**修正**:
1. ツール生成時に `DoyalistApproach` テーブルへ自動保存（仮想プロジェクト経由）
2. `/api/doyalist/approaches` API でタブ別に取得
3. 履歴ページの空表示を「ツールを作成すると履歴がここに出ます」に

**学び**: 「UI上は動いてる」と「データ的に正しい」は別。**フロント+バックエンド両方を確認する。**

### ⑧ URL未取得時に何も表示しないと混乱
**症状**: 多くの企業（特に小規模）はgBizINFOに公式URLが登録なし。「公式サイト」ボタンが出ない企業がほとんど。

**真の原因**: 業界別の取得率は実測で **0〜10%**（gBizINFO は政府調達・補助金受給企業中心のDB）。

**修正**: URL未取得時に「🌐 登録URLなし」と明示表示。データの透明性を保つ。

**学び**: **「データがない」も正直にUIに出す**ことが信頼につながる。隠すと「壊れてる？」と思われる。

### ⑨ 「メールサポート」が選択肢になっていた
**症状**: 料金プランに「メールサポート」「優先サポート」と書いてあった。

**問題**: 1人運営なのに「サポート充実」を謳ったらクレームになる。

**修正**: 料金プランから該当文言を削除。

**学び**: AIは「BtoB SaaSっぽい文言」を勝手に書く。**実態に合わない文言は明示的に削除**を依頼する。

### ⑩ 大量抽出時のVercelタイムアウト
**症状**: 10,000社抽出 + 詳細情報取得を全部やろうとして、Vercelの300秒制限を超えた。

**真の原因**: 詳細API（gBizINFO の `/hojin/{番号}`）を 10,000回 × 並列8 で叩くと約4分かかる。

**修正**:
1. 詳細取得を **並列12** に増加
2. wall-clock 予算 240秒を設定し、超過したら以降の取得を諦める
3. 取得できなかった企業は「登録URLなし」表示

**学び**: クラウド関数（Vercel・Lambda等）には**実行時間制限**がある。長時間処理は分割するか早期中断機構が必須。

## 8. 次に作るなら気をつけたいこと

### ✅ 最初にやるべきこと
1. **業界を知る**: 同じ分野のSaaS（Sales Marker・Musubu等）を 3〜5個触ってメモを取る
2. **データの実態調査**: 使うAPIを実際に叩いて、上限・形式・取得率を測る
3. **モックアップを画像で作る**: ChatGPTやFigmaで先に画面イメージを作る
4. **MVPを最小で**: 「これだけあれば動く」最小機能だけで一度本番リリース

### ⚠️ よくある落とし穴
1. **「あったら便利」を最初から全部実装しない** → 沼にハマる
2. **「公式ドキュメントを信用しすぎ」** → 実測で確認（ドヤリストでは limit/page の組み合わせが盲点だった）
3. **「セキュリティは後で」** → 後でやると地獄。最初から
4. **「テストは省略」** → 自動テストは省略OK。**ただし手動で本番触る**は絶対
5. **「料金プランを高く設定」** → 売れなくなる。最初は安く
6. **「全部1人でやろうとする」** → 困ったら他のサービス（Stripe・Vercel等）に任せる

### ✨ 文系ならではの強み
1. **ユーザー目線が強い**: コードよりUXに集中できる
2. **ビジネス感覚**: 「これ売れるか」を最初に考えられる
3. **言葉で伝えるのが得意**: Claudeへの指示が明確になる
4. **完璧主義に陥らない**: 「動けばOK」と割り切れる
