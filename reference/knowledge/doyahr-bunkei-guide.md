## 1. そもそもドヤHRとは何か

### 一言でいうと
**「従業員情報・組織図・人事評価を、ひとつの画面でまとめて管理できる中小企業向けの人材管理ツール」** です。

### 何ができるか
- **従業員データベース**: 顔写真・部署・役職・入社日・連絡先を一元管理（CSVで一括登録もできる）
- **組織図**: 登録した部署・上長関係から組織ツリーを自動生成
- **人事評価（MBO）**: 評価期間をつくり、メンバーごとに「自己評価 → 上司評価 → 最終評価」を回す。**AIが評価コメントの下書き**も作る
- **ダッシュボード**: 従業員数・部署数・進行中の評価などをひと目で把握
- **権限管理**: オーナー / 管理者 / メンバーの3段階。**他社のデータは絶対に見えない**組織単位のデータ分離
- **課金**: 無料プラン + 有料プラン（Stripe決済、ドヤAI共通の統一課金）

### 開発期間と工数
- 着手からローンチ準備完了まで: **実働 数日**（土台のMVPは別セッションで先行）
- 修正・改善のラリー: **約40回**（バグレポート → 修正のループ）
- 作業した人: **1人（事業担当・コード書けない）**
- 主に使ったAI: **Claude（Anthropic社）**

### 何が「すごい」のか
- 普通なら「3人 × 4ヶ月 = 1,200万円」レベルのHR SaaSを、AI活用で**1人 × 実働数日 = ほぼ実費のみ**で実現
- 顔写真付き従業員DB・組織図・MBO評価・AIコメント生成・Stripe決済・組織別データ分離まで含めた**実用レベル**
- バグが出ても **「修正してください」+ エラー本文** で次々解消

### NextAuth Google OAuth の設定（必須・自分でやる）
ドヤHRは Google ログイン必須なので、これだけは事前準備が必要:

1. https://console.cloud.google.com/ で新規プロジェクト作成
2. 「APIとサービス」→「認証情報」→「OAuth 2.0 クライアント ID 作成」
3. 承認済みリダイレクト URI に `https://your-app.vercel.app/api/auth/callback/google` を追加
4. 取得した Client ID / Secret を Vercel の環境変数に設定:
   - `GOOGLE_CLIENT_ID`
   - `GOOGLE_CLIENT_SECRET`
   - `NEXTAUTH_SECRET`（ランダムな32文字）
   - `NEXTAUTH_URL`（本番URL・末尾スラッシュなし）

これは Claude に頼んでも代わりにやってもらえないので、自分でやる必要あり。

### 心構え
1. **「1つの軸」から完成させる**。ドヤHRは「従業員管理」を土台にして、そこに組織図・評価を足した
2. **動くものを最速で作る → そこから直していく** が鉄則
3. **バグが連続報告されても焦らない**。1つずつ解消すれば必ず終わる
4. **エラーの本文・HTTPステータス（404 / 405 / 500）を丸ごとClaudeに貼る** が最強のデバッグ手段
5. **1回の依頼は1つのまとまった機能に絞る**。大きく頼むほど抜け漏れが増える

## 3. 開発の全体の流れ

ドヤHRは「5フェーズ」で進めました。

| フェーズ | ゴール | あなたがやること |
|---|---|---|
| ① 要件定義 | 何を作るか言語化 | 困りごと・登場人物・必須機能を箇条書きで渡す |
| ② MVP | 動く最小版 | 「まず従業員の登録・一覧だけ」と範囲を絞って依頼 |
| ③ 作り込み | 機能追加・UX改善 | 1機能ずつ依頼。毎回ブラウザで触って確認 |
| ④ 品質チェック | 実データで踏み込みテスト | 実際に数名登録して全ボタンを押す。不具合を正確に報告 |
| ⑤ ローンチ | 公開・課金有効化 | 本番設定（決済キー等）と公開ステータス変更を依頼 |

### Day 1: 設計と土台（従業員管理）

#### Step 1. 「何を作るか」を Claude に説明
ドヤHRの場合、こう伝えました:

> 「中小企業（5〜100名）向けのタレントマネジメント（人材管理）SaaSを作りたい。
> 従業員情報がExcelでバラバラ、人事評価がメールで煩雑、という困りごとを解決したい。
> 登場人物は オーナー（全権）/ 管理者（編集可）/ 一般メンバー（自分の情報のみ）。
> まず欲しいのは従業員の登録・一覧・編集。将来は組織図・人事評価・CSV一括登録。
> Googleログイン必須。**まず設計（画面・データ項目・API）の案を出して**」

> 💡コツ: **「まず設計案を出して」**と頼むと、いきなり実装に走らず全体像をすり合わせられる。

#### Step 2. データベース設計（Claudeに任せる）
「組織・従業員・部署・評価期間・評価の Prisma スキーマを作って。
**組織ごとにデータが完全に分離される**設計で、全テーブルに `organizationId` を入れて」

→ Claude が組織・従業員・部署・評価まわりのモデルを設計。
**ポイント**: 全テーブルに `organizationId` があるか必ず確認する（他社データ漏洩の防止）。

#### Step 3. 認証・基本ページ
- NextAuth で Google OAuth ログイン
- 組織の自動作成ロジック（初回ログインで自分の組織ができる）
- サイドバー付きレイアウト
- ダッシュボード / 従業員一覧 / 従業員詳細 / 設定

### Day 2: メイン機能（組織図・人事評価）

#### Step 4. 組織図
- 部署と上長関係から組織ツリーを自動描画

#### Step 5. 人事評価（MBO）
- 評価期間の作成（ドラフト → 公開 → 締め）
- メンバーごとに 自己評価 → 上司評価 → 最終評価
- **AIが評価コメントの下書きを生成**

#### Step 6. CSV一括登録
- テンプレCSVをダウンロード → 記入 → アップロードで従業員を一括作成

### Day 3: 仕上げとローンチ

#### Step 7. 共通項目に揃える（重要）
ドヤHRは「ドヤAIシリーズ」の1つなので、**バラバラに作らず既存の共通の仕組みに合わせた**:
- サイドバーを全サービス共通のものに統一
- 通知（トースト）を共通のものに統一
- 料金プランを全サービス共通の「無料 / プロ」2プランに統一

> 詳しくは [⑤ 共通項目に揃える依頼](#共通項目に揃える依頼重要) を参照。

#### Step 8. バグ修正のラリー
- 本番にデプロイして実際に触る → 不具合をスクショ + 本文で報告 → 「修正してください」（約40回）

- Stripe を本番キーに揃える
- 公開ステータスを「準備中」から「公開（active）」に変更
- GitHub に push すると Vercel が自動で本番反映

## 5. ドヤHR開発でハマった12のポイントと解決法

### ① 評価期間の詳細が「見つかりません」（405エラー）
**症状**: ダッシュボードの評価期間カードを押すと「見つかりません」。

**真の原因**: 詳細ページが `GET /api/.../periods/{id}` を叩いていたが、そのAPIに **GETハンドラが無く**、更新用の `PATCH` しか実装されていなかった。だから405（メソッド未許可）が返っていた。

**修正**: 同じAPIファイルに `GET` ハンドラを追加（評価件数も一緒に返すように）。

**学び**: **404と405は意味が違う**。404＝ページ/URLが無い、405＝URLはあるがそのメソッド（GET/POST等）が無い。**ステータスコードを見るだけで原因がほぼ確定する。**

---

### ② 新規作成直後のデータで画面がクラッシュ
**症状**: 評価フォームや1on1フォームを新規で開くと、白画面・エラーになることがあった。

**真の原因**: 作成直後は「目標リスト」「アジェンダ」などの配列項目が **null（空っぽ）** で返る。それを `.length` で参照していたのでクラッシュした。

**修正**: `(リスト?.length ?? 0) > 0` のように、**null を安全に扱う書き方**に統一。引数のデフォルト値（`= []`）だけでは null は防げない点に注意。

**学び**: **「新規作成直後の空データ」は最大のテストケース**。「作った瞬間に壊れないか」を必ず確認する。

---

### ③ 従業員の「編集」を押すと404
**症状**: 従業員詳細から「編集」に飛ぶと「ページが見つかりません」。

**真の原因**: 編集ページ自体が作られていなかった（リンクだけ先にあった）。

**修正**: 編集ページを新規作成（既存データを読み込んで埋める → 保存でAPIにPATCH）。

**学び**: **一覧⇄詳細⇄編集のリンク先が全部存在するか**を最後に通しで確認する。「飛んだ先が無い」は地味に多い。

---

### ④ 評価期間の進捗が「0/4」のまま動かない
**症状**: 評価期間の詳細で、評価が進んでいるのに「0 / 4」と表示され続ける。

**真の原因**: APIは結果を `items` というキーで返しているのに、画面側は `evaluations` というキーで読もうとしていた。**キー名の食い違い**で常に空だった。

**修正**: 画面側を正しいキー（`items`）に修正。ステータスの数え方もダッシュボードと揃えた。

**学び**: **APIが返すJSONの「実際のキー名」を目視で確認**する。想定と実物がズレていることは多い。

---

### ⑤ エラーメッセージが英語のまま出た
**症状**: 同じ評価を重複作成しようとすると、英語のエラー文がそのまま画面に出た。

**修正**: サーバー側のエラーメッセージを日本語に変更。

**学び**: AIが書くエラー文は英語になりがち。**ユーザーに見える文言は日本語か**を明示的にチェックして直す。

---

### ⑥ アップグレード（決済）が全部500エラー
**症状**: プロにアップグレードしようとすると 500。エラー本文に
`No such customer ... a similar object exists in test mode, but a live mode key was used` と出た。

**真の原因**: Stripeの **テストモードで作った顧客ID** に、**本番モードのキー** でアクセスしていた。テスト/本番の環境が混ざっていた。

**修正**:
1. キー（テスト/本番）と、顧客・商品・価格データの環境を**必ず揃える**
2. コード側は「顧客IDが見つからなければ作り直す」**自己修復**にしておく（環境が混ざっても落ちにくくなる）

**学び**: **Stripeはテスト世界と本番世界が完全に別物**。`sk_test_`/`pk_test_` と `sk_live_`/`pk_live_`、Price ID、Webhook、顧客データを「全部テスト」か「全部本番」で揃える。

---

### ⑦ 料金が画面によってバラバラ
**症状**: 料金ページと設定画面で、表示される金額が違った（¥14,800 と ¥9,980 など）。

**真の原因**: 価格を複数のファイルに**直書き**していたので、画面ごとにズレた。

**修正**: 価格は**1か所（単一ソース）**で管理し、全画面がそこを参照する形に統一。さらに「全サービス共通の料金UI部品」に寄せて、各画面で金額を書かないようにした。

**学び**: **同じ数字を複数箇所に書かない**。1か所に集約して全画面が参照する（DRY原則）。

---

### ⑧ 「プロ」「スタンダード」ボタンを押しても進まない
**症状**: 料金ページのボタンを押しても何も起こらない。

**真の原因**: ボタンのリンク先が仮の `#` のまま放置されていた（実装途中の名残）。

**修正**: 共通の料金UI部品（実際に決済へ進む）に置き換え。

**学び**: **仮リンク（`#`）や未実装のボタンが残っていないか**を洗い出す。「押せるのに進まない」は仮実装の置き忘れが定番。

---

### ⑨ ボタンが見づらい（コントラスト不足）
**症状**: 料金ボタンの文字色と背景色が近くて読みにくかった。

**真の原因**: 共通ボタンの既定スタイル（グラデーション + 白文字）と、上から当てた色がケンカしていた。

**修正**: 無料ボタンは白地に濃い文字、プロボタンは目立つ配色 + 読みやすい文字色に調整。

**学び**: **「動く」と「見やすい」は別問題**。実機で見て、コントラスト（文字と背景の差）を確認する。

---

### ⑩ 使わない機能（1on1）が中途半端に残っていた
**症状**: 「1on1履歴」などの項目が画面のあちこちに残っていた。

**修正**: まず**非表示**にして様子を見て、不要が確定してから消した。

**学び**: いったん使わない機能は、**いきなり消さず先に非表示**にすると、要件が戻ったときの手戻りが小さい。

---

### ⑪ 「修正したのに直っていない」とすれ違う
**症状**: Claudeが「修正完了」と言っても、実機では直っていないことがあった。

**真の原因**: デプロイ反映前に触っていた／ブラウザに古い画面がキャッシュされていた／既存の不正データが残っていた。

**修正**: デプロイ反映を数分待ってから確認。ハードリロード。バージョン表示で「今どの版か」を確認できるように。

**学び**: **「コード修正 ≠ ユーザー体験の修正」**。デプロイ反映・キャッシュクリアまでセットで考える。

---

### ⑫ 並行作業でファイルが消えかけた
**症状**: 複数のAIセッションを同じプロジェクトに同時に走らせて、変更が消える事故が起きた。

**修正**: **1プロジェクト = 1セッション**に統一。

**学び**: 同じフォルダを複数のAIで同時編集しない。取り合いで上書き・消失が起きる。

## 6. 文系でも分かる用語集

| 用語 | 平たく言うと |
|---|---|
| **タレントマネジメント** | 従業員の情報・スキル・評価をまとめて管理して活かすこと |
| **MBO** | 目標管理。「目標を立てて → 評価する」人事評価のやり方 |
| **組織図** | 誰がどの部署で、誰の下にいるかを表したツリー図 |
| **マルチテナント** | 1つのシステムを複数の会社で使い、互いのデータは見えない仕組み |
| **organizationId（組織ID）** | 「このデータはどの会社のものか」を表す印。これでデータを分ける |
| **IDOR** | URLのID（例 `/employee/123`）を `124` に変えて他人の情報を見れてしまう脆弱性 |
| **RBAC（権限管理）** | オーナー / 管理者 / メンバー など、役割ごとに「できること」を変える仕組み |
| **API** | 「データちょうだい」と頼むと返してくれる裏側の窓口 |
| **GET / POST / PATCH** | APIへの頼み方の種類。GET=取得、POST=作成、PATCH=一部更新 |
| **404 / 405 / 500** | エラーの種類。404=URLが無い、405=そのメソッドが無い、500=サーバーが落ちた |
| **null（ヌル）** | 「中身が空っぽ」の状態。新規作成直後に起きやすく、クラッシュの原因 |
| **Prisma** | データベースを TypeScript で扱える道具（ORM） |
| **NextAuth** | Googleログインなどを簡単に実装するライブラリ |
| **Stripe** | クレジットカード決済の代行サービス |
| **テスト/本番モード（Stripe）** | 練習用の世界と本物の世界。混ぜると決済が壊れる |
| **Webhook** | Stripe側の出来事（決済成功など）をこちらに自動通知する仕組み |
| **トースト** | 画面の隅に出る「保存しました」などの小さな通知 |
| **単一ソース（DRY）** | 同じ情報（価格など）を1か所だけに書くこと |
| **デプロイ / キャッシュ** | 公開作業 / 古い画面を一時保管して表示してしまう仕組み |

## 8. 次に作るなら気をつけたいこと

### ✅ 最初にやるべきこと
1. **既存のHRツールを触る**: SmartHR / カオナビ / freee人事労務 などをユーザー目線で観察
2. **データモデルから先に決める**: 後から大幅変更すると面倒
3. **「1つの軸」を完成させてから足す**: ドヤHRは従業員管理が土台。そこに組織図・評価を増築
4. **Google OAuth は最初に設定**: 後回しにすると詰む
5. **組織分離（organizationId）を最初から**: 他社データが見える事故は致命的

### ⚠️ よくある落とし穴
1. **新規作成直後のnullで画面が壊れる** → 配列は空配列で初期化、`?.` で安全に参照
2. **ルーティングの抜け（404/405）** → 一覧・詳細・編集・各APIのメソッド（GET/POST等）が全部揃っているか確認
3. **価格の二重定義** → 1か所に集約。各画面で金額を直書きしない
4. **決済のテスト/本番モード不整合** → 鍵と商品・顧客データの環境を必ず揃える。コードは自己修復に
5. **仮リンク（#）の置き忘れ** → 「押せるのに進まない」を最後に洗い出す
6. **並行セッション** → 1プロジェクト=1セッション

### ✨ Claude を使う時のコツ
1. **連続報告には番号を付ける**: 「① ② ③」で構造化すると修正漏れが減る
2. **エラーの本文 + ステータス（404/405/500）をそのまま貼る**: 原因がほぼ確定する
3. **「まず調べて理解した内容を報告して」**: 思い込み実装を防げる
4. **「ビルドが通ることを確認して」**: 完了報告の品質が上がる
5. **「共通項目に合わせて、独自実装しないで」**: バラバラ実装を防ぐ
6. **「消さずに一旦非表示にして」**: 要件が戻ったときの手戻りを最小化
7. **「正直に：できていない所はそう言って」**: 確認できていない箇所を炙り出す

### 💡 ドヤHR特有のノウハウ
1. **顔写真中心の従業員DB**: 文字の一覧より「顔が並ぶ」だけで一気に使いやすくなる
2. **CSV一括登録**: 初期導入の手間が激減する。テンプレCSVをダウンロードさせる形が親切
3. **MBO評価は「期間 → メンバー → 3段階評価」の入れ子**: 状態（ドラフト/公開/締め）を持たせると運用しやすい
4. **AI評価コメント**: ゼロから書くのは大変なので「下書きをAIが作る」だけで価値が高い
5. **組織単位のデータ分離は絶対**: 全クエリに organizationId。個別取得は必ず「自分の組織のものか」を確認

## まとめ

**「コードが書けない=作れない」は、もう古い時代の話です。**

Claude のような AI アシスタントを「優秀な開発パートナー」として活用すれば、
文系・非エンジニアでもドヤHRレベルの実用的な人材管理SaaSが、実働数日で作れます。

ただし大事なのは:
1. **1つの軸から完成させる**（ドヤHRは従業員管理が土台）
2. **エラーの本文とステータス（404/405/500）を丁寧にClaudeに伝える**
3. **「修正済み」を鵜呑みにせず実機テスト**（デプロイ反映・キャッシュに注意）
4. **共通項目に揃える**（サイドバー・通知・課金をバラバラに作らない）
5. **組織分離（organizationId）と権限は最初から**

これだけでプロダクトの品質は大きく上がります。**まずは小さく始めてみてください。**
