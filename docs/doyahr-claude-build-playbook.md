## 1. DoyaHR とは（完成イメージの共有）

DoyaHR は中小企業（5〜100名）向けのタレントマネジメント（人材管理）ツールです。主な機能：

- **従業員データベース**（顔写真・部署・役職・入社日・連絡先、CSV一括登録）
- **組織図**（部署ツリーの自動生成）
- **人事評価**（MBO：評価期間 → メンバーごとに自己評価／上司評価／最終評価、AIによる評価コメント生成）
- **ダッシュボード**（従業員数・部署数・進行中の評価などの集計）
- **権限管理**（オーナー／管理者／メンバー）と**組織単位のデータ分離**
- **課金**（無料プラン＋有料プラン、Stripe決済）

> 💡ポイント：最初から全部を狙わず、**「まず1つの軸（例：従業員管理）」を完成**させ、
> そこに評価・組織図…と足していくと崩れません。DoyaHR も従業員管理が土台でした。

## 2. 全体の進め方（5フェーズ）

| フェーズ | ゴール | あなたがやること |
|---|---|---|
| ① 要件定義 | 何を作るか言語化 | 業務の困りごと・登場人物・必須機能を箇条書きで渡す |
| ② MVP | 動く最小版 | 「まず従業員の登録・一覧だけ」など範囲を絞って依頼 |
| ③ 作り込み | 機能追加・UX改善 | 1機能ずつ依頼。毎回ブラウザで触って確認 |
| ④ 品質チェック | 実データで踏み込みテスト | 実際に数名登録し、全ボタンを押す。不具合を正確に報告 |
| ⑤ ローンチ | 公開・課金有効化 | 本番設定（決済キー等）と公開ステータス変更を依頼 |

**鉄則：1回の依頼は「1つのまとまった機能・修正」に絞る。** 大きく頼むほど抜け漏れが増えます。

## 3. ① 要件定義のプロンプト

最初に「作りたいものの全体像」を渡します。完璧でなくてOK。Claudeが質問してくれます。

### 📋 そのまま使えるプロンプト（要件定義）

```
[業種・会社規模] 向けの社内[人材管理／勤怠／日報…]ツールを作りたいです。
解決したい困りごと：
- [例] 従業員情報がExcelでバラバラ
- [例] 人事評価のやり取りがメールで煩雑
登場人物（役割）：
- オーナー（全権）／管理者（編集可）／一般メンバー（自分の情報のみ）
最初に絶対欲しい機能（MVP）：
- [例] 従業員の登録・一覧・編集
将来足したい機能：
- [例] 組織図、人事評価、CSV一括登録
デザインの希望：
- [例] 既存サービスと統一感を持たせたい／日本語UI／ポップで見やすく
不明点があれば質問してください。まず設計（画面・データ項目・API）の案を出してください。
```

> 💡コツ：**「まず設計案を出して」**と頼むと、いきなり実装に走らず全体像をすり合わせられます。

## 4. ②③ 実装を依頼するプロンプト

範囲を絞って依頼します。「作って終わり」ではなく「作って、ビルドが通ることまで確認して」と添えると品質が上がります。

### 📋 そのまま使えるプロンプト（機能実装）

```
[機能名]を実装してください。
- 画面：[一覧／詳細／作成フォーム…]
- データ項目：[氏名・部署・役職…]
- 権限：[管理者のみ作成可、など]
- 完了条件：ビルド（コンパイル）が通ること、実際に画面で操作できること
既存の作法（命名・デザイン・API構成）に合わせてください。
```

### 「共通項目に揃える」依頼（重要）

複数サービス／既存システムがある場合、**バラバラに作らせない**のが肝心です。DoyaHRでも、
サイドバー・通知（トースト）・課金を「全サービス共通の仕組み」に揃える指示で一気に整いました。

```
このツールのUI/課金は、既存サービスの「共通項目」に合わせてください。
まず共通の仕組み（共通サイドバー、共通の通知、共通の料金プラン）を調べて理解し、
独自実装せずそれらを使う形にしてください。理解した内容を先に報告してください。
```

> 💡「**まず調べて理解した内容を報告して**」を挟むと、思い込み実装を防げます。

## 5. ④ テスト・不具合報告のしかた（ここが品質を決める）

Claudeは「報告された不具合」を直すのが非常に得意です。**良い不具合報告＝速い修正**。

### 効く不具合報告の型

```
【症状】[何をしたら／どの画面で] [何が起きたか]
【再現手順】1. … 2. … 3. …
【期待】本来こうなってほしい
【手がかり】（あれば）画面のエラー文・コンソールのエラー・通信(API)の状態
```

実際にDoyaHRで「効いた」報告の例：

- 「評価期間カードを押すと『見つかりません』。`GET /api/.../periods/{id}` が**405**を返している。
  GETハンドラが無いのでは？」 → 原因特定が一発で、即修正。
- 「アップグレードが500。エラー本文に
  `No such customer ... a similar object exists in test mode, but a live mode key was used` と出る」
  → テスト/本番モードの不整合と判明、即対処。

> 💡コツ：**エラーの「本文」やHTTPステータス（404/405/500）をそのまま貼る**と、原因がほぼ確定します。
> 開発者ツール（F12）の「Console」「Network」タブを見るだけでOK。

## 6. ⑤ ローンチ前チェックリスト

DoyaHRのローンチ準備で実際に確認した項目です。Claudeに「この観点でゼロから最終チェックして」と頼むのも有効。

- [ ] 主要導線が全部動く（作成→一覧→詳細→編集→削除）
- [ ] **新規作成直後**のデータでも画面が壊れない（空配列/未入力での表示）
- [ ] 一覧⇄詳細⇄編集の**リンク先が全部存在**する（「ページが見つかりません」が出ない）
- [ ] 権限：他人のデータが見えない／編集できない
- [ ] エラー時に**日本語**で分かるメッセージが出る（英語のまま放置しない）
- [ ] 料金表示が**全画面で一致**（複数箇所に価格を直書きしていないか）
- [ ] 決済：**テスト/本番モードのキーと商品・顧客データが揃っている**
- [ ] 公開ステータスを「準備中／メンテ」から「公開（active）」に変更
- [ ] 不要な機能は**消すのではなく非表示**にして、後で戻せるようにする

## 7. つまずきやすいポイント（DoyaHR実例つき）

実際に踏んだ落とし穴です。先回りで「ここ大丈夫？」と聞くと事故が減ります。

1. **新規データのnullで画面クラッシュ**
   作成直後は配列項目（目標リスト等）が空(null)で返り、`.length`参照でエラー。
   → 「新規作成直後のデータでも壊れないように、配列は空配列で初期化して」

2. **ルーティングの抜け（404/405）**
   一覧から「編集」へ飛ぶのにそのページ／API（GET）が無く404・405。
   → 「一覧・詳細・編集・各APIの**メソッド（GET/POST等）が全部揃っているか**確認して」

3. **価格・設定の二重定義**
   料金を複数ファイルに直書きして、画面ごとに金額がバラバラに。
   → 「価格は**1か所（単一ソース）**で管理し、全画面がそこを参照する形にして」

4. **決済のテスト/本番モード不整合**
   テストモードで作った顧客IDに本番キーでアクセスして500。
   → 「**鍵（テスト/本番）と、商品・価格・顧客データの環境を必ず揃える**」。
     コード側は「無ければ作り直す」自己修復にしておくと安全。

5. **ボタンが“押せるのに進まない”**
   仮実装でリンク先が `#` のまま放置され、押しても無反応。
   → 「**仮リンク（#）や未実装のボタンが残っていないか**洗い出して」

6. **並行で複数のAIセッションを同じプロジェクトに走らせない**
   ファイルの取り合いで変更が消える事故が起きます。**1プロジェクト＝1セッション**を推奨。

7. **削除より「非表示」**
   いったん使わない機能は、コードごと消さず非表示に。要件が戻ったときの手戻りが小さい。

## 9. 技術メモ（依頼相手＝開発担当者向けの参考）

DoyaHR の構成（同種ツールの“型”として）：

- フロント：Next.js（App Router）＋ React ＋ TypeScript、Tailwind CSS、Framer Motion
- データ：PostgreSQL ＋ Prisma（`organizationId` 列で組織ごとにデータ分離）
- 認証：NextAuth（Googleログイン）／役割（オーナー・管理者・メンバー）
- 課金：Stripe（**全サービス共通の価格ID・統一プラン**に寄せると保守が楽）
- AI：評価コメント生成・要約に LLM を利用
- 通知：共通トースト（`react-hot-toast`）、デザインは共通サイドバー／テーマに統一

> このメモは「同じ作法で増築する」ための地図です。新機能も既存の作法に合わせると、
> 画面・API・課金がバラけず、長く保守できます。
