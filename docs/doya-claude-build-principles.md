# Claudeでうまく開発するための指針 ― ドヤシリーズ開発の黄金律

> 社外配布用。エンジニアでない方でも、Claude（Claude Code）に依頼してSaaS／業務ツールを
> 作るときに「**こう進めればうまくいく / こう外すと失敗する**」を、実開発（DoyaHR・ドヤスライド・
> ドヤ営業管理）から抽出した普遍ルールとしてまとめた指針書です。
> サービス別の具体手順は各「開発ナレッジ（プレイブック）」を、全体像は「開発内容まとめ（スライド用）」を参照。

---

## この指針の前提

- あなたの役割は **「発注者 兼 テスター」**。コードはClaudeが書く。
- あなたの成果物は **「正確な日本語」**。何を作るか・どこが今おかしいかを、的確に伝える。
- うまくいくかどうかは、**伝え方と確認の丁寧さ**でほぼ決まる。

---

## 1. 進め方の黄金律（5原則）

### 原則1：小さく刻む（1依頼＝1機能）
大きく頼むほど抜け漏れる。「まず◯◯だけ」と範囲を絞る。
> ✅「1機能ずつお願いします。今回は商談の一覧表示だけ」
> ❌「営業管理ツール一式を作って」

### 原則2：作る前に設計させる
いきなり実装に走らせない。まず**画面・データ・APIの設計案**を出させて、全体像をすり合わせる。
> ✅「まず設計（画面・データの関係図・API）の案を出して。不明点は質問して」

### 原則3：調べさせてから着手させる
思い込み実装を防ぐ最強のひと言。
> ✅「まず既存の作法を調べて、**理解した内容を報告してから**着手して」

### 原則4：完了条件を必ず添える
「作って終わり」にしない。
> ✅「**ビルドが通ること**と、**実際に画面で操作できること**を確認してから完了報告して」

### 原則5：触って、正確に報告する
品質は不具合報告の質で決まる。症状・再現手順・期待・手がかり（エラー本文／HTTPステータス）を貼る。
> ✅「3枚目だけ色が違う。再現は…。期待は…。コンソールに `500` が出ている」

---

## 2. 設計の黄金律（最初に固めるべき4つ）

最初に決めずに進めると、後で**全画面の作り直し**になる項目。

| 観点 | 指針 | 依頼の言い方 |
|---|---|---|
| **データの骨格** | 登場する「モノ」と関係を最初に図にする | 「データの関係図を先に設計して」 |
| **データ分離** | 誰のデータか（組織/ユーザー）で必ず分ける | 「組織IDで分離し、他者のデータは見えなくして」 |
| **権限（役割）** | 役割の段階と昇格防止を最初に決める | 「役割は◯>◯>◯。権限の昇格防止も入れて」 |
| **共通の型** | UI・課金・認証は独自実装せず共通に揃える | 「共通の仕組みを調べて、それに合わせて」 |

> 💡「データ分離」と「権限」は**後付けが最も危険**。最初に握る。

---

## 3. 品質を上げる「効く言い回し」集

- 「**まず調べて、理解した内容を報告してから**着手して」
- 「**ビルドが通ること**を確認してから完了報告して」
- 「**1機能ずつ**お願いします。今回は〇〇だけ」
- 「既存の**共通項目（デザイン・課金・認証）に合わせて**、独自実装しないで」
- 「この不具合を**根本原因から**直して（対症療法でなく）」
- 「**ゼロベースで最終チェック**して、ローンチ前に直すべき重大な不具合を挙げて」
- 「消さずに**一旦非表示**にして。あとで戻せるように」
- 「**正直に**：できていない所・確認できていない所はそう言って」

---

## 4. 失敗を防ぐ7つの定番リスク（先回りで潰す）

実開発で実際に踏んだ落とし穴。**「ここ大丈夫？」と先に聞くだけで事故が激減**します。

1. **新規データ（空＝null）で画面が壊れる**
   → 「新規作成直後でも壊れないよう、配列は空配列で初期化して」
2. **一覧→編集のページ／APIが無くて 404・405**
   → 「一覧・詳細・編集・各APIのメソッドが全部揃っているか確認して」
3. **価格・設定の二重定義で表示がバラバラ**
   → 「価格は1か所で管理し、全画面がそこを参照する形にして」
4. **決済のテスト/本番モード不整合で500**
   → 「鍵と商品・顧客データの環境を揃える。無ければ作り直す自己修復に」
5. **押せるのに進まないボタン（仮リンク `#` 放置）**
   → 「仮リンクや未実装のボタンが残っていないか洗い出して」
6. **複数AIを同じプロジェクトに同時起動 → 変更が消える**
   → **1プロジェクト＝1セッション**を厳守。
7. **いらない機能を“削除”して後で手戻り**
   → 削除ではなく**非表示**。戻せるようにしておく。

---

## 5. ローンチ前の最終チェック（共通版）

- [ ] 主要導線が全部動く（作成→一覧→詳細→編集→削除）
- [ ] 新規作成直後（データ空）でも画面が壊れない
- [ ] 一覧⇄詳細⇄編集のリンク先が全部存在する（404が出ない）
- [ ] データ分離：他者／他組織のデータが見えない・触れない
- [ ] 権限：他人のデータを編集できない／権限を勝手に上げられない
- [ ] エラー時に日本語で分かるメッセージが出る
- [ ] 料金表示が全画面で一致（価格は1か所管理）
- [ ] 決済のキーと商品・顧客データの環境が揃っている
- [ ] 公開ステータスを「準備中／メンテ」から「公開（active）」に変更
- [ ] 不要な機能は消さず非表示

---

## 6. これだけは覚えて帰る ― 3行サマリー

1. **小さく刻んで、設計を先に、調べさせてから着手させる。**
2. **データ分離と権限と共通の型は、最初に固める。**
3. **触って、エラー本文ごと正確に報告する。**

> コードが書けないことは、もう参入障壁ではない。
> 「正しく伝える」スキルがあれば、SaaSは立ち上がる。
