## 1. ドヤスライドとは（完成イメージの共有）

ドヤスライドは「**テーマを入力するだけで、AIが全スライドを1枚絵のフル画像として生成する**」スライド作成ツールです。主な機能：

- **テーマ→構成→画像生成**（資料タイプ・テーマ・枚数・比率・カラー・スタイルを選ぶだけ）
- **ロゴ自動合成**（最初に上げたロゴを各スライドの指定位置に合成して統一感）
- **チャットでその場修正**（「2枚目の文字を大きく」等を会話で指示して再生成）
- **バージョン履歴で巻き戻し**（やり直しても前の版に戻せる）
- **12種のスタイル**（コーポレート〜ド派手）と**PNG(ZIP)/PDF書き出し**
- **課金**（無料プラン＋有料プラン、全サービス共通の統一プラン）

> 💡ポイント：このツールの心臓は「**画像生成**」です。だからこそ
> 「①構成を作る（文章）」と「②画像にする（絵）」の**2段階に分けて作る**と崩れません。
> ドヤスライドも『構成生成→画像生成』の分業がうまくいった鍵でした。

## 2. 「このように作ればうまくいく」― 成功の設計指針

画像生成ツールならではの、最初に握っておくべき指針です。**ここを外すと後で必ず作り直しになります。**

1. **生成は2段階に分ける**
   「テーマ→**構成テキスト**（各ページの見出し・本文・絵の指示）→**画像化**」。
   いきなり画像から作らせない。文章の骨組みを先に固めると、絵がブレません。

2. **画像生成の窓口は1つに統一する**
   モデル（gpt-image-2 等）を直接あちこちで呼ばない。**共通の1関数を必ず経由**させる。
   こうしておくと「モデルを変える」「不調時に別モデルへ自動切替」が1か所の修正で済みます。
   → 依頼：「画像生成は**必ず共通の窓口（ディスパッチャ）経由**にして。各画面から直接APIを叩かないで」

3. **スタイルは“見本”を事前に焼いておく**
   12種のスタイルは毎回ゼロから生成すると遅い・高い。**代表ページの見本を1回作ってキャッシュ**し、選択画面はそれを見せる。
   → 依頼：「スタイル選択のプレビューは**共有キャッシュ**にして、毎回生成し直さないで」

4. **色は1か所で決め、全ページで統一する**
   ページごとに色がバラつくのが一番ダサい。「**デッキ全体で配色を固定**」をルール化。
   → 依頼：「1つの資料の中は**テーマカラー＋ニュートラル**に統一して。ページごとに色を変えないで」

5. **重い処理は時間制限に注意**
   画像を何枚も作ると時間がかかる。**並行生成**＋**1枚ずつ再生成できる**設計に。
   → 依頼：「全ページを**並行で生成**して。1枚だけ作り直す再生成ボタンも付けて」

| フェーズ | ゴール | あなたがやること |

| ① 要件定義 | 何を作るか言語化 | 資料の用途・スタイル・出力形式を箇条書きで渡す |

## 4. ① 要件定義のプロンプト

### 📋 そのまま使えるプロンプト（要件定義）

```
AIで[営業資料／提案書／SNS投稿…]を自動生成するスライドツールを作りたいです。
コンセプト：テーマを入力したら、AIが各スライドを画像として生成する。
入力してほしい項目：資料タイプ・テーマ・補足・枚数・比率(16:9/1:1/9:16)・カラー・スタイル・ロゴ
ほしい体験：
- 生成後にチャットで「ここを直して」と修正できる
- やり直しても前の版に戻せる（履歴）
- 最後にPNG（ZIP）とPDFで書き出せる
デザインの希望：日本語UI / 既存サービスと統一感
不明点があれば質問してください。まず設計（画面・データ項目・API・生成フロー）の案を出してください。
```

> 💡コツ：**「生成フローも設計に含めて」**と頼むと、構成生成→画像化の2段階を最初から織り込んでくれます。

## 6. ④ テスト・不具合報告のしかた（品質を決める）

画像系は「**目で見て**おかしい」を伝えるのが大事です。文章だけでなく、何枚目のどこが、を具体的に。

### 効く不具合報告の型

```
【症状】[何枚目／どの画面で] [何が起きたか]（例：3枚目だけ色が他と違う／文字が枠からはみ出す）
【再現手順】1. … 2. … 3. …
【期待】本来こうなってほしい
【手がかり】（あれば）エラー文・HTTPステータス・生成に何秒かかったか
```

画像ツールで「効く」報告の例：

- 「**スタイルを変えても見本が同じ**。プレビューのキャッシュが更新されていないのでは？」
- 「8枚生成すると**途中でタイムアウト**する。並行生成 or 枚数上限の見直しを」
- 「資料の中で**ページごとに色がバラバラ**。配色を全ページ統一する指示が効いていない」

> 💡コツ：見た目の問題は「**他のページと比べてどう違うか**」を書くと直しやすい。

## 7. ⑤ ローンチ前チェックリスト

- [ ] テーマを入れて**最後まで（構成→画像→出力）**通る
- [ ] **全スタイル**でそれらしい見本が出る（プレビューが古くない）
- [ ] ロゴを上げると**各ページの指定位置に合成**される／位置・サイズ変更が効く
- [ ] チャット修正・**履歴の巻き戻し**が動く
- [ ] PNG(ZIP)・PDFの**両方**が正しく書き出せる
- [ ] 1資料の中で**配色が統一**されている
- [ ] 無料/有料の**生成枚数の上限**が効く（課金は統一プラン）
- [ ] 画像生成は**共通窓口経由**（直接API呼びが残っていない）
- [ ] 公開ステータスを「準備中」から「公開（active）」に変更
- [ ] 公式ロゴ画像の配置（無いとフォールバック表示になる）

## 8. つまずきやすいポイント（ドヤスライド実例つき）

1. **モデルを直接あちこちで呼んで保守不能**
   画像APIを各画面から直接叩くと、モデル変更時に全部直す羽目に。
   → 「**画像生成は共通の1関数経由**に統一。各所の直接呼び出しを無くして」

2. **スタイル見本が更新されない**
   プレビューをキャッシュしているのに、プロンプトを変えても古い見本のまま。
   → 「プレビューの**キャッシュの版（バージョン）を上げて**全部焼き直して」

3. **多ページ生成でタイムアウト**
   直列に1枚ずつ作ると、枚数が多いと制限時間を超える。
   → 「全ページを**並行生成**に。重い処理の時間制限（maxDuration）も確認して」

4. **資料内の配色がバラバラ**
   各ページの絵の指示に色を書くと、ページごとに色が変わる。
   → 「**1資料＝1配色**で固定。絵の指示の中の色指定は無視させて」

5. **特別ページの誤判定**
   「表紙」「まとめ」だけ特別レイアウトにしたいのに、本文まで特別扱いされる。
   → 「役割名の**完全一致**でだけ特別ページにして（部分一致で誤判定しない）」

6. **並行で複数AIセッションを同じプロジェクトに走らせない**
   ファイルの取り合いで変更が消えます。**1プロジェクト＝1セッション**。

7. **削除より「非表示」**
   旧スタイルや旧機能はコードごと消さず非表示に。戻したくなったときの手戻りが小さい。

## 10. 技術メモ（依頼相手＝開発担当者向けの参考）

ドヤスライドの構成（同種ツールの“型”として）：

- フロント：Next.js（App Router）＋ React ＋ TypeScript、Tailwind CSS
- データ：PostgreSQL ＋ Prisma（プロジェクト／スライド／資産／チャット／バージョンの5種で管理、`userId` で分離）
- **画像生成**：必ず共通ディスパッチャ経由（メイン＝gpt-image-2／不調時フォールバック）。比率に応じてサイズ自動切替
- **構成生成**：テキストLLM（Gemini系）で各ページの見出し・本文・絵の指示を作る
- ストレージ：生成画像・ロゴ・スタイル見本をクラウドストレージに保存（見本は共有キャッシュ）
- 課金：**全サービス共通の統一プラン**（プラン判定は1か所＝ユーザーのプラン列を参照）
- 重い処理：全ページ並行生成＋1枚再生成。実行時間の上限に注意

> このメモは「同じ作法で増築する」ための地図です。画像系は特に
> **生成窓口の一本化・配色統一・キャッシュの版管理**を最初に決めておくと、長く保守できます。
